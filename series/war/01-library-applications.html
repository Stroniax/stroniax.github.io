A Library of Applications

PSSharp Series PowerShell Tutorial Library of Applications

PowerShell is like a library of applications. If you're like me, sometimes you go to the library and you just browse. Let's start out by browsing the PowerShell library. Run the following code in your PowerShell terminal. 

Get-Command

You're going to see a whole lot of commands scroll by. Each command in PowerShell can be thought of as an application. I was overwhelmed by that information - let's look in the library for a specific application - like using a catalogue to find a single book. Run the following code in your PowerShell terminal. 

Get-Command -Name 'gcm'

Let's take a look at the output of of our command. 

CommandType     Name                                               Version    Source
-----------     ----                                               -------    ------
Alias           gcm -> Get-Command

We have a table of information, presented as text, telling us about the commands that are available. That's like looking in the library catalogue and seeing the books they have!

The result we see is a command, which is a tool that we can run in PowerShell. Much like an application we can run it, but instead of opening a window, most commands only run in the terminal.

There are a few different types of commands. We want to be accurate with our terminology, but this is more technical than you probably need right now; feel free to come back and read this section again later.

  • Script

    A script is a PowerShell file (ending in .ps1) which can be run by PowerShell. It's the simplest way to make a reusable group of PowerShell code. You can run a script by typing the full name of the script file in the terminal, e.g. ./path/to/script.ps1.

  • Function

    A function, like a script, is a bunch of PowerShell code that runs together. Instead of running a file, you can define or load a function in PowerShell and it'll stick around until you close the window.

    Functions are generally easier to use because you don't have to worry about where the file is located, but there are some considerations to take into account which we'll take about in the next article.

  • Cmdlet

    A cmdlet is a special type of function that is written in a dotnet language like C# (or F# or Visual Basic) and compiled into a DLL. Because they can't be written directly in PowerShell they're a little more complicated to write, but they tend to be more consistent, reliable, and performant.

    I'm writing a series on how to learn C# by writing PowerShell cmdlets that you can check out after this series.

  • Application

    An application is an executable program (on Windows that's a file ending in .exe) that PowerShell can run. Like a command prompt, PowerShell can run these. They're more portable that the other PowerShell command types because they generally don't require PowerShell to run, but they are less useful in PowerShell that the other types of commands because they only accept text input and output. While we see things as text in PowerShell, what we are dealing with often enough are actually rich objects with descriptive values called properties and ways to interact with those values called methods. Because applications only support text input/output (to be compatible with non-PowerShell use), it's preferable to use the other types of commands whenever possible.

So there are different types of a command, but from a use perspective they're pretty much the same. You type the name of the command, and you hit enter. The command does something - maybe it changes a setting, maybe it displays information. Let's create a command!
